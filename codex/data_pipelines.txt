Exactly.

Imagine

```

```

The app ships with them.

No API.

Fast.

---

But...

I wouldn't use TXT.

I'd use JSON.

For example

```json
{
  "id": "prob_001",
  "difficulty": "medium",
  "topic": "Expected Value",
  "question": "...",
  "latex": "...",
  "answer": "...",
  "hint": "...",
  "tags": ["Probability","Betting"]
}
```

JSON is much easier to extend.

Tomorrow you add

* hints
* references
* difficulty
* sources
* images

Nothing breaks.

Chapter 1
Content Philosophy

This belongs at the top.

Rules like:

Every piece of content must have a purpose.
Quality is more important than quantity.
Content should encourage thinking, not memorization.
The user should learn principles rather than isolated facts.
Every question should create curiosity.
Every article should invite reflection.

Chapter 10
Future Expansion

Hermes should be able to support future Item Types without changing the pipeline.

Examples

Research Papers

Books

Experiments

Projects

Code Snippets

Flashcards

Case Studies

The pipeline should remain the same.

Only the Item Type changes.

Bro...

I think we just found another architectural law.

Content should be modular. The application should never care whether an Item is a Question, Article, Quote, or something invented five years later.

That's how Hermes stays scalable.
